# Добавить источник вакансий

Задача: по URL страницы вакансий написать адаптер в `sources.local/`, протестировать
на живых данных и подключить. Пользователю ничего делать не нужно.

## Интерфейс адаптера

Один файл `sources.local/<имя>.mjs`:

```js
export async function fetchFoo(cfg) {
  // cfg — весь config.json; свои параметры клади в cfg.foo (см. примеры)
  return [{
    source: "foo",        // короткое имя, оно же ключ в config.sources
    id: "123",            // стабильный id вакансии на сайте (для дедупа)
    title: "…",           // название вакансии — по нему работает фильтр
    company: "…",
    url: "https://…",     // прямая ссылка на вакансию (проверь, что открывается!)
    salary: "",           // человекочитаемо: "200k–300k ₽" или ""
    location: "",         // "Москва · удалёнка" или ""
    published: "",        // ISO-дата или ""
  }];
}
export default { name: "foo", fetch: fetchFoo };
```

Автодискавери подхватит файл сам. В `config.json` → `sources` добавь `"foo": true`.
Если источник фильтрует сам (например, по полному тексту) — ставь у элементов
`prefiltered: true`, иначе центральный фильтр отфильтрует по `title` повторно.

## Как найти данные (по убыванию частоты, примеры — в examples/sources/)

1. **Официальный API или RSS** — проверь первым: `/api/...`, `/vacancies/rss`,
   документацию. Примеры: `examples/sources/yandex.mjs`, `src/sources/habr.mjs`.
2. **Скрытый JSON API SPA** — curl'ни страницу; если она пустая оболочка, ищи
   эндпоинты: (а) в HTML/JS-бандлах: `grep -oE '"/[a-z/]*api[^"]*"'`, конфиг-объекты
   `*_API`, `__NEXT_DATA__`, buildId; (б) перебором типовых путей `/api/vacancies`.
   Смотри у ответа реальные параметры фильтров и пагинации — они часто с
   нестандартными именами (`direction_ids[]`, `take/skip`, `page_size`, курсор).
   Примеры: `examples/sources/alfa.mjs`, `tochka.mjs`, `rwb.mjs`.
3. **Данные вшиты в SSR-HTML** — grep страницы по названию вакансии с сайта;
   если есть — парси регэкспами вшитый JSON. Пример: `examples/sources/gis2.mjs`.
4. **AJAX с особыми заголовками** — попробуй `X-Requested-With: XMLHttpRequest`,
   Referer, POST формы. Пример: `examples/sources/avito.mjs` (Bitrix).
5. **Перехват из браузера** — если есть браузерные инструменты (Claude in Chrome
   или Browser pane): открой страницу, смотри
   `performance.getEntriesByType("resource")` с фильтром по api/vacanc/graphql,
   меняй фильтры на странице, вылавливай запрос; тело POST ищи в JS-бандлах.
   Пример добытого так: `examples/sources/tbank.mjs`.

## Когда сдаться (и что сказать пользователю)

Если сайт за бот-протекшеном (редирект-петля, JS-челлендж, F5/DDoS-Guard,
SSR-only выдача только браузерам) — НЕ строй обход. Скажи прямо, что прямой
адаптер невозможен, и проверь, публикует ли компания вакансии на hh.ru — почти
все публикуют (тогда её закрывает источник `hh` с токеном). Известные тупики
уже описаны в CLAUDE.md — не трать время на них повторно.

## Чек-лист перед завершением

1. `curl`/`node` запрос работает БЕЗ браузерных кук (адаптер будет жить в кроне).
2. Ссылка `url` из результата открывается (HTTP 200) и ведёт на вакансию.
3. Пагинация: если вакансий больше страницы — обойди все (с `await new Promise(r
   => setTimeout(r, 300))` между запросами), с потолком страниц.
4. Ошибки — бросай `throw new Error("<источник> HTTP <код>")`: оркестратор
   покажет сбой в дайджесте, а не уронит прогон.
5. Прогони `node src/collect.mjs` — источник должен появиться в логе с числом
   вакансий; проверь пару ссылок из дайджеста.
6. Если у источника есть свой фильтр (направление/категория) — вынеси слаг в
   `config.json` (см. как это сделано у примеров) и покажи пользователю, где менять.
